# Big update [RU](/ru/changelogs/9545)/[EN](/en/changelogs/9545)

This update has two main focus areas:

- a new Blazor UI and the gradual move away from the old WPF shell — [build 9339](/changelogs/9339), [build 9406](/changelogs/9406)
- built-in [AI Assistant](/features/ai-assistant), [Discord Bot](/features/discord-bot), and [Codex Coding Assistant](/features/ai-codex-coding-assistant)

This is probably the biggest EyeAuras update by sheer volume of changes across all the years of development. Even with testing, an update this large can still surface issues. If something broke or behaves strangely, please send a report through **Report a problem** in the app header. I do read those reports and investigate them.

One important user-facing AI change: there are now free limits, so you can try the Assistant without manually setting up an OpenAI account. This is especially important for users in Russia, because OpenAI currently blocks access for them, and EyeAuras Gateway solves that problem. If you want full control, you can provide your own OpenAI API key or a compatible endpoint; in that mode, EyeAuras connects directly and does not use the Gateway.

Other major areas in this update:

- almost all Free-version limitations have been removed; one notable exception is still the Network Message trigger/action, and I plan to remove that leftover part of the old model in one of the next major updates
- fully redesigned [pack licenses]( /features/sublicenses) and sublicenses for pack and [mini-app](/features/mini-app) authors — you can now build almost any licensing model you need from the available options, for example tracking session count or binding access to a computer
- [EyePad](/features/eyepad) as a separate workflow for code, `.csx`, `.sln`, [Live Import](/scripting/ide-integration), and external IDEs
- stronger [C# scripting](/scripting/ide-integration): [Embedded Resources](/scripting/embedded-resources), [NuGet](/scripting/nuget), [Memory API](/scripting/memory-api/getting-started), Script Properties, and IDE integration
- improvements to [Export / Import](/features/export-import), pack publishing, and [mini-app](/features/mini-app) distribution workflows

If you are new to EyeAuras: a [pack](/features/packs) is a collection of auras/scripts, while a [mini-app](/features/mini-app) is a complete solution built on top of EyeAuras.

## AI integration

The second main focus of this update is AI. It is now split into two usage scenarios:

- [AI Assistant](/features/ai-assistant) — a smart reference and a lightweight mode for small edits to settings, documentation, and simple examples
- [Codex Coding Assistant](/features/ai-codex-coding-assistant) — a much more powerful tool that can effectively write a program _for_ you

![AI Assistant](https://s3.eyeauras.net/media/2026/03/EyeAuras_DzRWqPXjls.png =x600)

### Tokens, API keys, free limits, and Gateway

EyeAuras integrates with OpenAI and OpenAI-compatible APIs: you can provide your own OpenAI API key or compatible endpoint, and requests will go there directly.

OpenAI does **not** provide free limits for this kind of third-party integration, but I still wanted users to be able to try the new functionality without extra setup. That is why EyeAuras handles this through `EyeAuras Gateway` — essentially a proxy server in front of OpenAI. Thanks to Gateway, free limits are available for both Assistant and Codex: you can get started without a separate OpenAI account, keys, or endpoints. That should be enough for occasional AI help.

Gateway also solves the access restrictions for users in Russia: OpenAI currently blocks direct access for them, while EyeAuras Gateway lets them use AI features without setting up their own routing workarounds. If you provide your own API key or endpoint, Gateway is not used. In the near future, there will also be a way to top up an EyeAuras balance so users in Russia can continue using AI more fully.

### AI Assistant as a smart reference

[AI Assistant](/features/ai-assistant) is a smart EyeAuras reference and a lightweight mode for simple changes: which trigger to choose, why an action did not fire, where the right wiki section is, what a minimal C# script looks like, or what to fix in the current setup.

Alongside it, [EyeAuras Bot](/features/discord-bot) is now available in Discord. It offers the same quick-help format outside the app: the bot knows the wiki, changelog, and scripting best practices.

![EyeAuras Bot In Discord](https://s3.eyeauras.net/media/2026/04/Discord_jjGZGKtG1r.png =x600)

The wiki is now available in both Russian and English. This matters both for users and for AI: the better the documentation is, the more accurate Assistant, Discord Bot, Codex, and external coding agents become.

### Codex Coding Assistant and how to choose

[Codex Coding Assistant](/features/ai-codex-coding-assistant) is the stronger mode for tasks where a simple answer or small edit is not enough. It works alongside [EyePad](/features/eyepad) and external IDEs: it can see the files and folders in the workspace, keep task context, make a plan, apply edits, take editor diagnostics into account, and explain what changed.

![EyeAuras Codex](https://s3.eyeauras.net/media/2026/04/EyeAuras_N9sVSOeQ5s.png =x600)

Typical Codex scenarios: writing or fixing a C# script, understanding Behavior Trees, building a pack, preparing a mini-app, using [Live Import](/scripting/ide-integration), connecting the [Memory API](/scripting/memory-api/getting-started), or explaining an error directly in the current project.

Codex is intended to become the main AI mode in EyeAuras for tasks where the whole project matters.

In short: Assistant is cheaper and faster for reference, ideas, or a simple edit. Codex is smarter and better suited for “help me do this in my project”.

## Monetization

The old Free-version restrictions have been almost completely removed: limits on aura count, script size, integrations, and core capabilities should no longer be constrained by the old Free/Pro model.

![Free becomes Pro](https://s3.eyeauras.net/media/2025/12/msedge_LIoQ4IG1h8.png =x600)

The one notable exception is still network messages: using the [Network Message trigger](/triggers/network-message) and [Send Network Message action](/actions/send-network-message) still requires a license. This is a technical leftover from the old model, and I plan to remove it in one of the next major updates.

Overall, EyeAuras is now moving toward a free core application. Monetization should shift away from restrictions inside the app itself and toward finished products built on top of it: packs, mini-apps, and custom author-made solutions.

## Sublicenses for pack and mini-app authors

[Sublicenses](/features/sublicenses) are no longer about restricting EyeAuras itself as a program. They are about how authors can distribute their own solutions.

If you build a [pack](/features/packs), [mini-app](/features/mini-app), or private automation for a team or client, you can now assemble a more suitable access model from the available options. For example:

- a personal license for one computer
- a license for multiple concurrent sessions
- separate access for a team or client
- limited demo/trial access before purchase
- different access levels for the basic and extended version of a pack

The idea is simple: EyeAuras should stay as open as possible as a platform, while authors can still monetize what they build on top of it cleanly.

## Memory API

The [Memory API](/scripting/memory-api/getting-started) now has its own documentation section. It is a way to read state directly from a process. The main use case is C# scripts, but you can also pass read values through variables if needed.

For example, you can read coordinates, HP, the current target, object lists, and similar data.

What this gives you in practice compared to using WinAPI directly:

- the same script can work with different memory sources: a [regular process](/scripting/memory-api/backends/local-process), [MemProcFS / LeechCore](/scripting/memory-api/backends/lc-process), your own driver, a separate device, or an external bridge such as reading through DLL inject. In other words, you can start with something simple like `LocalProcess`, then change literally one line and use the same code with a DMA device, driver, or your own highly custom read method
- if you already have your own read method, you can plug it in through [custom process](/scripting/memory-api/backends/custom-process) or [custom memory backend](/scripting/memory-api/backends/custom-memory-backends), while keeping the same API and all the surrounding features
- when you need to run code inside the target process, there are [code caves](/scripting/memory-api/code-caves), allocation/protection/free memory, [DLL inject](/scripting/memory-api/dll-inject), APC, remote thread, and manual mapping — essentially everything you need for comfortable work
- [EA ReProcess pack](https://eyeauras.net/share/S20260422083348SIiNglgRSqSJ) is an open-source example of building a full RE tool on ImGui around the Memory API; the [ReProcess API docs](/scripting/api/reverse-engineering/reprocess) describe the workspace, tabs, actions, address inspectors, and knowledge storage
- `ReToolsPlugin` can be connected directly inside your own Memory API scripts to give an AI agent tools for reversing the selected process: reading memory, searching data, disassembling, parsing RTTI, saving findings, and continuing from discovered facts

## EyePad

[EyePad](/features/eyepad) continues evolving into a separate workflow mode for people who write code, run scripts, import packs, and work with `.sln`.

![EyePad Live Import](https://s3.eyeauras.net/media/2026/03/EyeAuras_ehBoMmbNoO.png =x600)

The important shift here is that [EyePad](/features/eyepad) is now closer to a real working environment rather than just an embedded editor. You can open standalone `.csx` and text files, work with `.sln`, run [Live Import](/scripting/ide-integration), import a [pack](/features/packs), and gradually turn an experiment into a full project.

File handling reliability has also been improved. Drafts, saved files, external disk changes, and restoration after restart should now behave more predictably. This is especially important for AI workflows: if [Codex](/features/ai-codex-coding-assistant) or an external agent is working with a project, it needs a real file-based context rather than “magic” text living only in editor memory.

This is not the final Explorer/workspace model yet, but the direction is already clear: EyePad should become a proper shell for the workflow of “write code, run it immediately, and turn it into a pack or mini-app”.
