# New Release

It has been a few months since the last stable release (~15 weeks), so this update includes quite a lot of changes.

Short version:
- new website
- reworked publish/subscribe functionality
- server-side packaging — a huge feature and, essentially, the infrastructure that lets you build your own mini-apps on top of EyeAuras functionality
- a LOT of scripting improvements and changes. WebUI has not been migrated to the new scripting engine (v3) yet; that will be one of the goals of the next release
- many, many changes to behavior trees — they are still relatively new, but at this point you should at least consider using them instead of the default aura mechanism, because they usually simplify things a lot after the initial learning period
- enabling conditions were reworked and are now much more stable
- nearly a dozen different crashes were fixed, and around 40 JIRA tasks were closed

My main focus was server-side packaging and the website infrastructure around it. I hope this becomes the main way you distribute your mini-apps, because with a few more improvements it can reduce the setup process for new users to just a few clicks.

## New Feature — Server-Side Packaging

Any shared aura pack can now be downloaded as a standalone portable application that runs in parallel with the main EyeAuras app. This should greatly reduce the learning curve for new users — to them, your aura pack will look like a regular program they can download, unpack, and run.

This becomes especially powerful when combined with [Custom UI](/en/overlays/custom-ui) — you can build an entirely new application and distribute it as a portable app that still benefits from multiple layers of anti-cheat protection, nearly a dozen different input simulators, high-performance image capture, and machine learning capabilities.

This is still in its early stages, but it is meant to become the recommended way to distribute your work to other users, including people who already use EyeAuras. The portable format also guarantees that the app will keep working even if the main EyeAuras version gets a breaking update.

P.S. Client-side packaging will be enabled a bit later.

## Rework — Publish/Sync

This is an improved version of the aura import mechanism. If you provide a link to a pack, for example [Blum crypto game clicker](https://eyeauras.net/share/S20240606142306qxWxc7fy40p6), you will start receiving update notifications as soon as the author releases a new version.

There is also a built-in system for merging your local settings with the author's pack settings.

The goal of this feature is to make distributing updated aura packs easier for authors, and updating them easier for users. In the near future, pack settings will also allow authors to specify a recommended program version, and the update mechanism will be able to download and install it.

### Merging Local Pack Configuration with the Remote Version

For example, imagine you subscribed to an aura that activates a specific window when you press F4. It uses the HotkeyIsActive trigger together with the WinActivate action.

In version 1, the author set the key to `F4` and the window title to `MyGame`.

You subscribe to the pack and download it. However, `F4` is inconvenient for you, so you change it to `F3`.

Later, the author updates the pack and adds another action. You receive an in-app notification that an update is available.

When you click **Update**, the merge system analyzes what changed.

In this case, it sees:
- Local changes (yours): **HotkeyIsActive** — key changed from `F4` to `F3`
- Remote changes (author's): a new action was added

These changes do not conflict, so the merge system can create a unique intermediate version that includes both the author's changes and your local changes.

A conflict can also happen. For example, the author may decide to change the key as well (`F4` => `F2`). In that case, the system detects conflicting settings. Right now, conflicts are always resolved in favor of the author's settings.

On the `Changes` tab, you can click `Download` at any time to see how your local settings differ from the author's current pack. No changes will be applied — this is only a preview.

## Scripting Improvements and Bug Fixes

### Cross-Aura References Added

This long-awaited change lets you link multiple auras that contain C# code and reuse methods, classes, and everything else as if it were a single script.

This means you can write utility code once and reuse it whenever needed. Hopefully, this will drive copy-paste in larger projects close to zero.

This applies to both auras and behavior trees — if you reference an aura from a behavior tree, its shared code becomes available to all nodes in that tree.

To link auras together, just add a Reference, as shown below.

![Code references](https://s3.eyeauras.net/media/2024/06/tSL7ye5wGTRZCCo1.png)

#### How can cross-auras help me?

For example, you can keep a set of classes in your "library" aura that store user configuration, simulate input in a custom way, or draw something on the screen via OnScreenDisplay.

### Ability to Write Your Own Mouse Input Smoothers — [read more](/en/scripting/examples/advanced/custom-input-smoother)

Input smoothers are used to make mouse movement... well... smooth. There are many different algorithms, and EyeAuras currently includes only two of them.

In this release, I made it possible to implement your own smoother in scripts and then use it anywhere — in behavior trees, in auras, or in other scripts.

The next step I want to take is allowing entire input simulators to be implemented in code. That way, you could create an input method that sends input to a background window, for example. I want to add one myself, but new tasks keep piling up, so exposing this capability should unblock people who can code it on their own.

### Handling Critical Exceptions in User Scripts

Previously, when a script did something `really` wrong — the kind of thing that would normally cause a **crash** in a regular application — EyeAuras behaved the same way.

That meant if you made a mistake in a script, the entire program could crash as soon as it happened, opening the error report window. That made some sense early on, but at this point I am getting a LOT of error reports that I cannot fix, because they need to be fixed `in the script itself`.

Basically, there are 3 options:

- `a)` Leave everything as it is and keep telling people they should implement exception handling practices like regular applications do.  
  This does not work very well, especially with the growing number of new users. The more popular the program becomes, the more new users will try scripting, and the more of them will hit exceptions. Not very scalable.

- `b)` Run scripts in an isolated environment — essentially small separate executables that communicate with the main program through some elegant transport.  
  This is the best option, but technically extremely difficult, especially since that "small" program would need to communicate with the main app with extremely low latency or it simply would not be usable. At some point this will become one of my goals, but I estimate the cost at roughly 3–4 months of work at minimum, which makes it one of the most expensive potential new features.

- `c)` Since EyeAuras fully controls the source code and the way it executes, make the entire scripting process as robust and easy to use as possible.  
  This is the option we are trying now. I built a test solution that will `try to catch all errors` coming from user scripts and, instead of crashing the application, try to handle them, restore state where possible, and continue running. I will also start adding automated code improvements that are applied to user code during compilation — for example, injecting proper exception handling where it is missing, such as button click handlers, or handling exceptions raised from tasks.

We'll see how it goes.

A few other fixes and improvements in this release:
- [Bug fix] Previously, the internal system that releases services created via `GetService<T>` could sometimes release (i.e. kill) important EyeAuras services
- [Improvement] Compilation times should be shorter, especially for the first compile after startup
- [Improvement] The mechanism that runs scripts now has much lower internal latency, which means your scripts can run much more often — this optimization was done mainly for [Behavior Trees](/en/behavior-trees/gettings-started), since a single cycle may execute dozens of small script nodes

### Intercept Conditions

These have actually existed for years, but they did not get enough attention even though they are extremely useful in some scenarios.

In general, this mechanism directly controls whether hotkey handling is enabled at any given moment. By linking one or more auras, you can define the exact set of conditions that must be true before the trigger starts doing anything.

For example, if you link an aura that contains [WindowIsActive](/en/triggers/window-is-active), the trigger will only respond to key presses while the game window is active.

In more advanced cases, you can tie the trigger to some in-game state. For example, when you press RMB (part of the HotkeyIsActive configuration) **and** some powerful skill is off cooldown (linked condition), instead of using whatever is normally bound to RMB, you simulate another key that casts that skill.

A good example would be automating Vaal Skills in Path of Exile — you keep spamming the normal version of a skill, and as soon as the Vaal version has enough souls, it gets triggered automatically without you having to do anything.

To better explain the next two options, here is an example:

I have a HotkeyIsActive trigger listening for `F3` in `toggle` mode. That means the first press of `F3` `activates` the trigger, and a second press `deactivates` it. This is simple and very useful for turning more advanced auras on and off, such as auto-flasks.

I also enabled the `Suppress key` option, which blocks `F3` from reaching the game window — otherwise the game could react to whatever is bound to `F3`.

The problem with this setup is that if I leave it like this, `F3` will be blocked in all applications, not just in my game.

To fix that, I can add an `Intercept Condition` with [WindowIsActive](/en/triggers/window-is-active), which limits `F3` to that game window. That mostly works, but there are two issues with it:

#### Intercept Conditions — Ignore When the Hotkey Is Active

**Problem:**  
Now, to turn HotkeyIsActive off, I need the game window to be focused. So if I want to disable _something_ bound to `F3`, I first need to bring the game window to the front. If `F3` enabled some aggressive clicker, getting that window back into focus may be inconvenient.

**Solution:**  
When this option is enabled, **activation** of the trigger can still require the game window to be active, but **deactivation** can be done from anywhere.

Forgot to turn your clicker off before alt-tabbing? No problem — just press `F3` again and the trigger will be disabled.

Note that this option only has an effect when toggle mode is enabled.

#### Intercept Conditions — Automatically Deactivate Trigger

**Problem:**  
There is no fast way to disable HotkeyIsActive unless you press the button again. In some cases that is inconvenient — you have to remember that some automation is still running. You alt-tab away and forget about it. Then some trigger condition becomes true and you suddenly start doing things in the game. That can cause problems.

**Solution:**  
Just enable this new option. Now, if the trigger's intercept condition is no longer satisfied, the trigger is automatically deactivated.

In our example, if I alt-tab away from the game window, everything controlled by `F3` will be turned off. Note that you will need to enable it again after returning to the game.

This makes the whole system much more resistant to human error.

### New Option — Reset Trigger State When Linked Auras Are Inactive

A new option was added that resets trigger state (deactivates the trigger) whenever linked auras become inactive.

The most practical use case is a linked aura that includes the WindowIsActive trigger. By default, the behavior is:

- If the selected window is active, the trigger works normally and can be activated with a key press
- If the selected window is not active, the trigger does not intercept key presses. To deactivate the trigger, you first need to bring the selected window to the foreground

With the new option (`Reset trigger state when linked auras are inactive`), the behavior changes:

- If the selected window is active, the trigger works normally and can be activated with a key press
- If the selected window is not active, the trigger is automatically deactivated as soon as this state is detected

This makes it extremely easy to set up a key that enables some functionality only while the game is active, and automatically disables it when you alt-tab or minimize the game.

## Mouse Movement Verification Is Now Disabled

For several years, EyeAuras used a mechanism that checked the result of every mouse move — it verified that the cursor had actually moved to the intended location.

Historically, the main reason for this was hardware input emulators such as Usb2Kbd and Arduino-based devices. That made sense for them because they do not move the mouse instantly. If you had two inputs in a row (`MouseMove` + `Click`), there was a chance the mouse would not yet be in the correct place when the click happened.

Now that the tools for sending input have become much more powerful — SendSequence, BehaviorTrees, Scripts — this mechanism seems to cause more problems than it solves.

In **test** mode, this mechanism is now `disabled` for all input methods. In theory, this should not be noticeable in most cases, but keep in mind that you may need to add a bit of extra delay between mouse move operations.

### Aura Tree Improvements — Copy/Paste Support

Full Ctrl+C / Ctrl+V support has been added. Previously, Ctrl+C worked more like an export mechanism, which meant you could not simply copy an item and paste it into another folder because that could create conflicts.

Now you can copy and paste items throughout the tree, just like in Windows Explorer.

The naming scheme for cloned items now also follows the Windows style more closely. For example, if you copy `Aura`, the first copy will be named `Aura - Copy`, the second `Aura - Copy(2)`, and so on.

Note that copy/paste between multiple EyeAuras instances still does **not** work. For that, you still need to use Export and Import.

### Better UX for Image Capture Triggers

Historically, when editing image capture trigger settings (Image / Text / Color / ML), the preview window updated at the same `capture rate` you had configured, or lower if the trigger could not reach that FPS.

In most cases this was fine, but it became inconvenient once you started working with very low FPS values for efficiency reasons — for example, a trigger that updates `once` every `10 seconds`, or even `0 FPS` triggers used together with [C# scripts](/en/scripting/getting-started) and [Behavior Trees](/en/behavior-trees/gettings-started).

Previously, you had to click the `Refresh` button manually to force a preview update. Now you can set a global minimum preview FPS for the whole application, and that value will be used instead of the trigger's capture rate.

> Minimum preview FPS only applies while preview is enabled. It does not affect FPS afterward and is not exported as part of the trigger configuration.
