# Чтение списка сущностей из памяти для бота

| Поле | Значение |
| --- | --- |
| Сложность | Средняя |
| Пример | Демонстрация радара для Assault Cube, которая читает данные сущностей из памяти и показывает их в окне Blazor. |
| Конечная цель | Показать живой список сущностей из другого процесса с именами, позициями, здоровьем и другими похожими полями. |
| Ключевые технологии | Memory API, `RemoteMemoryObject`, память модуля, окно Blazor |

## Когда использовать

- Вы уже знаете целевой процесс и модуль.
- У вас есть offsets для указателя игрока, указателя списка сущностей, количества сущностей и нужных полей.
- Список сущностей достаточно небольшой, чтобы обновлять его по таймеру.
- Задача — просмотр данных, радар, отладка или простой вспомогательный UI.

## Пример

[Assault Cube 1.3.0.2 Radar](https://eyeauras.net/share/S20241111183049Hs524Q7FisJ1) — это очень маленький публичный pack с одной аурой `CsharpScript`. На странице pack он описан как демо Memory API вместе с BlazorWindow для отрисовки радара.

![Демо радара Assault Cube](https://s3.eyeauras.net/media/2024/11/ac_client_OCl6NDHXWXR9G0aC.jpg)

## Общая схема

- Открыть целевой процесс и создать память с привязкой к модулю.
- Описать корневой объект игры, локального игрока, список сущностей и сами сущности как классы-обёртки над памятью.
- Считать количество сущностей и указатель на список сущностей из известных offsets модуля.
- Пройтись по указателям сущностей и прочитать простые поля вроде имени, позиции, здоровья, брони, команды или типа.
- Показать текущий снимок данных в небольшом окне Blazor, таблице, радаре или логе.

## Что нужно решить заранее

- Какой backend процесса использовать: локальный процесс, MPFS/LC, KD или свой backend.
- Представлять сущности как живые объекты памяти или как неизменяемые снимки.
- Как часто обновлять список.
- Что делать с невалидными или `null`-указателями: пропускать, показывать как ошибки или останавливать обновление.
- Каким будет UI: таблица, радар, карта или отладочный лог.

## Что посмотреть в API

- `IProcess`
- `LocalProcess`
- `LCProcess`
- `KDProcess`
- `IMemory`
- `MemoryOfModule`
- `RemoteMemoryObject`
- `Read<T>`
- `ReadString`
- `IDialogWindowUnstableScriptingApi`
- `BlazorReactiveComponent`
- `BlazorReactiveComponent<T>`

## Чего лучше избегать

- Считать offsets стабильными между разными версиями игры.
- Читать все поля на каждом кадре, если на экране показывается только часть данных.
- Переносить математику указателей в UI-код.
- Падать на `null`, устаревших или невалидных указателях сущностей.
- Слишком рано превращать простой reader списка сущностей в полноценный bot framework.
